# 面向非程序人员的 AI 建站工具 —— 方案总览与可行性分析

> 文档版本：v0.1　|　最后更新：2026-06-12
>
> 关联参考实现：一个已采用 Nuxt 3 + Nuxt UI + GSAP + Lenis + Iconify 全套技术栈的真实项目，可作为脚手架原型

---

## 1. 背景与问题定义

### 1.1 现状

越来越多**没有技术背景的人**（运营、设计、产品、个人创作者）开始用 AI（Cursor / Claude / v0 / 各类 AI IDE）直接生成网站代码。在"单页面 / Demo"阶段体验很好，但随着页面增多、功能堆叠，会迅速暴露以下问题：

| 问题 | 根因 |
|------|------|
| 代码越来越乱 | 每次对话上下文不同，AI 生成风格不统一，没有约定的目录与组件规范 |
| 调试困难 | 出错时只有一句"白屏 / 报错"，缺乏可读的日志与定位手段 |
| 错误频出且反复 | 没有测试网，改 A 坏 B，AI 无法自我验证改动是否破坏功能 |
| 不敢迭代 | 没有"安全网"，每次改动都像拆盲盒 |

### 1.2 问题的本质

非程序人员遇到的并不是"不会写代码"，而是**缺少工程化的约束与反馈闭环**。

> 核心洞察：AI 生成代码的质量，高度依赖于**它所处环境给它的约束（护栏）和反馈（信号）**。
> 给 AI 一个混乱、无规范、无测试、无日志的项目，它只会越改越乱；
> 给 AI 一个有强约束、有自动反馈的项目，它能在护栏内稳定产出。

因此这个工具的真正价值不是"又一个技术栈模板"，而是：

> **一套让 AI 在护栏内工作、并能自动获得反馈的工程化脚手架（Guardrails + Feedback Loop for AI Coding）。**

用户提出的 4 个方向，恰好对应这套闭环的四个支柱：

```
统一架构  ──►  约束 AI 怎么写（Constraint）
日志系统  ──►  出错时能看见发生了什么（Observability）
测试体系  ──►  自动验证改动有没有破坏功能（Verification）
调试能力  ──►  快速定位并修复问题（Diagnosability）
```

## 2. 方案可行性分析

### 2.1 总体结论

**方案整体可行，方向正确，且 4 个维度高度互补、缺一不可。** 已有的参考实现项目证明该技术栈在真实业务中能跑通，将其抽象为脚手架是低风险、高收益的。

下面逐项分析。

### 2.2 维度一：统一架构（Nuxt 3 + Nuxt UI / Shadcn Vue + GSAP + Lenis + Iconify）

| 评估项 | 结论 |
|--------|------|
| 可行性 | ✅ 高。参考实现已用该套技术栈落地，社区成熟、文档完善 |
| 对非程序人员的友好度 | ✅ 高。Nuxt 约定式路由（`pages/` 即路由）、自动导入（composables/components 免 import）极大降低心智负担 |
| AI 友好度 | ✅ 高。技术栈越主流，AI 训练语料越多，生成质量越高 |
| 风险点 | ⚠️ Nuxt UI 3 与 Shadcn Vue 二选一需明确，避免两套并存导致风格分裂 |

**关键建议：**
- **UI 库只选一套并强约束**。推荐 **Nuxt UI 3**（与 Nuxt 深度集成、自带主题系统、上手最快）；若追求高度自定义视觉，再考虑 Shadcn Vue。两者不要混用。
- 必须配套一份 **AI 规则文件**（`.cursorrules` / `CLAUDE.md` / `AGENTS.md`），把"目录怎么放、组件怎么命名、用哪个库"写死，否则 AI 仍会自由发挥。这是统一架构能否真正生效的**决定性因素**。

> 详见 [`01-统一架构规范.md`](./01-统一架构规范.md)

### 2.3 维度二：完善的日志系统

| 评估项 | 结论 |
|--------|------|
| 可行性 | ✅ 高。Nuxt 同构特性允许在 Server（Nitro）和 Client 端统一日志方案 |
| 价值 | ⭐⭐⭐⭐⭐ 极高。对非程序人员，"看得见错误"是排障的前提 |
| 风险点 | ⚠️ 生产环境需控制日志量与脱敏；日志要"结构化"而非散落的 `console.log` |

**关键建议：**
- 提供一个统一的 `useLogger()` composable（封装级别、模块标签、结构化输出），**禁止裸用 `console.log`**。
- 客户端错误（未捕获异常、Vue 错误、Promise rejection）要**自动上报**到一个本地可见的面板，让非程序人员能直接复制错误发给 AI。
- 这一点是当前参考实现缺失的（`useApi.ts` 仅 `throw new Error`，无统一日志），属于本工具要补齐的核心能力。

> 详见 [`02-日志系统设计.md`](./02-日志系统设计.md)

### 2.4 维度三：单测 + 自动化测试（Playwright）

| 评估项 | 结论 |
|--------|------|
| 可行性 | ✅ 高。Vitest（单测）+ Playwright（E2E）是 Vue/Nuxt 生态标配 |
| 价值 | ⭐⭐⭐⭐⭐ 极高。测试是 AI 的"自动验收员"，让 AI 改完能自己验证 |
| 风险点 | ⚠️ 非程序人员不会写测试。需让"AI 写测试 + 一键运行 + 失败可读"成为默认工作流 |

**关键建议：**
- 定位调整：测试不是给非程序人员写的，而是**给 AI 写、给 AI 跑**的。工作流应为："你描述功能 → AI 写实现 + 写测试 → 自动运行 → 红灯则 AI 自我修复"。
- E2E 优先于单测：对建站场景，**Playwright 端到端"页面能否正常打开、按钮能否点击、流程能否走通"** 比函数级单测收益更直接。建议比例 E2E : 单测 ≈ 7 : 3。
- 提供 **冒烟测试模板**：自动遍历所有 `pages/` 路由，断言"无 JS 报错、无白屏、核心元素存在"。新增页面零成本获得基础保护。

> 详见 [`03-测试体系设计.md`](./03-测试体系设计.md)

### 2.5 维度四：完善的调试能力

| 评估项 | 结论 |
|--------|------|
| 可行性 | ✅ 高。Nuxt DevTools 开箱即用，可叠加自定义调试面板 |
| 价值 | ⭐⭐⭐⭐ 高。与日志系统协同，构成"发现问题→定位问题"链路 |
| 风险点 | ⚠️ 需对非程序人员"翻译"技术信息（友好错误页 + 一键复制诊断信息） |

**关键建议：**
- 提供**友好的错误页（`error.vue`）**：不暴露堆栈给终端用户，但提供"复制诊断信息"按钮，一键打包"错误 + 最近日志 + 环境信息"，供用户直接粘贴给 AI。
- 提供**应用内调试面板**（开发态浮层）：实时显示日志流、当前路由、状态、最近 API 请求。
- 接入 **Nuxt DevTools** 并默认开启。

> 详见 [`04-调试能力设计.md`](./04-调试能力设计.md)

## 3. 方案的补强：被忽略但关键的第五件事

仅有上述 4 项，AI 仍可能"自由发挥"。真正让 4 项落地的，是一条贯穿始终的主线：

### 3.1 ⭐ AI 协作规范（Guardrails as Code）

把所有约定写成 **AI 可读的规则文件**，放在项目根目录，让每次 AI 对话都自动遵守：

- `AGENTS.md` / `CLAUDE.md` / `.cursorrules`：技术栈、目录约定、命名规范、"必须做 / 禁止做"清单。
- 这是"统一架构"能否真正生效的开关。没有它，模板很快会被 AI 改回混乱状态。

### 3.2 一键脚手架与命令

非程序人员记不住命令。统一封装为极简脚本：

| 想做的事 | 一条命令 |
|----------|----------|
| 启动 | `npm run dev` |
| 新建一个页面 | `npm run new:page <名字>`（生成符合规范的页面 + 冒烟测试） |
| 跑全部检查 | `npm run check`（lint + 类型 + 单测 + E2E 冒烟） |
| 出问题打包诊断 | `npm run doctor`（收集日志/环境/报错为一个文件） |

### 3.3 推荐的最终能力地图

```
┌─────────────────────────────────────────────────────────┐
│                  AI 建站工具（脚手架）                      │
├───────────────┬───────────────┬──────────────────────────┤
│  ① 统一架构    │  ② 日志系统    │  ③ 测试体系     ④ 调试能力 │
│  Nuxt3+UI     │  useLogger    │  Vitest        Nuxt       │
│  GSAP/Lenis   │  错误自动上报   │  Playwright    DevTools   │
│  Iconify      │  结构化输出     │  路由冒烟模板    友好错误页 │
├───────────────┴───────────────┴──────────────────────────┤
│       ⑤ 贯穿主线：AI 协作规范（AGENTS.md）+ 一键命令         │
└─────────────────────────────────────────────────────────┘
                          ▲
                          │  反馈闭环
        描述需求 → AI 写代码+测试 → 一键检查 → 红灯则看日志自修复
```

## 4. 风险与权衡

| 风险 | 等级 | 应对 |
|------|------|------|
| 非程序人员被工程化概念劝退 | 中 | 所有能力都收敛为"一条命令 / 一个按钮"，技术细节隐藏 |
| AI 不遵守规范文件 | 中 | 规范文件简短、命令式、带正反例；并用 lint/CI 强制兜底 |
| 测试维护成本 | 中 | 以路由冒烟为主、由 AI 维护，不要求人工写 |
| Nuxt UI / Shadcn 选型反复 | 低 | 文档强约束二选一 |
| 过度工程化拖慢 Demo 速度 | 中 | 分级模板：`minimal`（仅架构）/ `standard`（+日志调试）/ `full`（+测试） |

## 5. 落地路线图（建议）

| 阶段 | 目标 | 产出 |
|------|------|------|
| P0 起步模板 | 把参考实现抽象为干净脚手架 | 可 `npx create` 的模板仓库 + `AGENTS.md` |
| P1 可观测 | 接入日志系统 + 友好错误页 | `useLogger` + `error.vue` + 调试面板 |
| P2 安全网 | 接入测试 | Vitest + Playwright + 路由冒烟模板 |
| P3 工作流 | 一键命令 + 脚手架生成器 | `new:page` / `check` / `doctor` |
| P4 打磨 | 文档、示例、模板分级 | 面向非程序人员的图文上手指南 |

## 6. 结论

- **方案可行，且 4 个方向选得准、互补强。**
- 建议补强**第 5 件事——AI 协作规范与一键命令**，它是让前 4 项真正生效的关键。
- 参考实现可直接作为 P0 抽象来源，落地风险低。
