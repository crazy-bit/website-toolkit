# 02 · 日志系统设计

> 目标：让"出错时看不见发生了什么"变成"出错时一眼能看见、并能一键复制给 AI"。
>
> 核心原则：**结构化、分级、统一入口、客户端错误自动捕获**。禁止裸用 `console.log`。

---

## 1. 设计目标

| 目标 | 说明 |
|------|------|
| 统一入口 | 全项目只用 `useLogger()`，便于统一格式与开关 |
| 分级 | `debug / info / warn / error`，生产可按级别过滤 |
| 结构化 | 每条日志含：时间、级别、模块、消息、上下文对象 |
| 同构 | Server（Nitro）与 Client 端共用一致的 API |
| 自动捕获 | 未捕获异常 / Vue 错误 / Promise rejection / 请求失败 全部自动记录 |
| 可导出 | 客户端日志缓存在内存环形缓冲，支持"一键复制最近 N 条" |
| 脱敏 & 限流 | 生产环境对敏感字段脱敏、对高频日志限流 |

---

## 2. 日志级别约定

| 级别 | 何时用 | 生产是否输出 |
|------|--------|--------------|
| `debug` | 开发期细节、变量值 | 否（仅开发） |
| `info` | 关键流程节点（页面加载、请求开始/成功） | 是（可配置） |
| `warn` | 可恢复异常、降级、用了默认值 | 是 |
| `error` | 功能失败、未捕获异常 | 是（必上报） |

---

## 3. 统一日志 API（`composables/useLogger.ts`）

设计成"带模块标签的工厂"，每个文件创建带自己模块名的 logger：

```typescript
// composables/useLogger.ts
type LogLevel = 'debug' | 'info' | 'warn' | 'error'

interface LogEntry {
  time: string
  level: LogLevel
  module: string
  message: string
  context?: Record<string, unknown>
}

// 内存环形缓冲：保留最近 200 条，供调试面板/一键复制使用
const buffer: LogEntry[] = []
const MAX = 200

const LEVEL_ORDER: Record<LogLevel, number> = { debug: 0, info: 1, warn: 2, error: 3 }

function push(entry: LogEntry) {
  buffer.push(entry)
  if (buffer.length > MAX) buffer.shift()
}

export function useLogger(module: string) {
  // 生产只输出 info 及以上，开发输出全部
  const minLevel: LogLevel = import.meta.dev ? 'debug' : 'info'

  function log(level: LogLevel, message: string, context?: Record<string, unknown>) {
    if (LEVEL_ORDER[level] < LEVEL_ORDER[minLevel]) return
    const entry: LogEntry = {
      time: new Date().toISOString(),
      level,
      module,
      message,
      context: sanitize(context),
    }
    push(entry)

    const tag = `%c[${level.toUpperCase()}]%c ${module}`
    const style = level === 'error' ? 'color:#fff;background:#ef4444;padding:1px 4px;border-radius:3px'
      : level === 'warn' ? 'color:#000;background:#f59e0b;padding:1px 4px;border-radius:3px'
      : 'color:#fff;background:#6366f1;padding:1px 4px;border-radius:3px'
    // eslint-disable-next-line no-console
    console[level === 'debug' ? 'log' : level](tag, style, '', message, context ?? '')

    // error 级别：可在此接入远端上报
    if (level === 'error') report(entry)
  }

  return {
    debug: (m: string, c?: Record<string, unknown>) => log('debug', m, c),
    info: (m: string, c?: Record<string, unknown>) => log('info', m, c),
    warn: (m: string, c?: Record<string, unknown>) => log('warn', m, c),
    error: (m: string, c?: Record<string, unknown>) => log('error', m, c),
  }
}

// 供调试面板 / 一键复制使用
export function getLogBuffer(): LogEntry[] {
  return [...buffer]
}

// 敏感字段脱敏
function sanitize(ctx?: Record<string, unknown>) {
  if (!ctx) return ctx
  const SENSITIVE = ['password', 'token', 'authorization', 'secret', 'phone', 'idcard']
  const out: Record<string, unknown> = {}
  for (const [k, v] of Object.entries(ctx)) {
    out[k] = SENSITIVE.some(s => k.toLowerCase().includes(s)) ? '***' : v
  }
  return out
}

function report(_entry: LogEntry) {
  // P1 可选：批量上报到 server/api/logs
}
```

### 用法（AI 生成代码时统一这样写）

```typescript
const log = useLogger('useApi')

log.info('开始拉取数据', { url: '/api/data' })
try {
  const data = await fetch('/api/data').then(r => r.json())
  log.info('数据拉取成功', { count: data.length })
} catch (e) {
  log.error('数据拉取失败', { error: String(e) })
  throw e
}
```

## 4. 客户端全局错误自动捕获（`plugins/error-capture.client.ts`）

非程序人员最怕的"白屏 / 没反应"，往往是未捕获异常。要全部兜住：

```typescript
// plugins/error-capture.client.ts
export default defineNuxtPlugin((nuxtApp) => {
  const log = useLogger('global')

  // Vue 组件渲染错误
  nuxtApp.vueApp.config.errorHandler = (err, _instance, info) => {
    log.error('Vue 渲染错误', { error: String(err), info })
  }

  // 未捕获的同步错误
  window.addEventListener('error', (e) => {
    log.error('未捕获错误', { message: e.message, source: e.filename, line: e.lineno })
  })

  // 未处理的 Promise rejection
  window.addEventListener('unhandledrejection', (e) => {
    log.error('未处理的 Promise 异常', { reason: String(e.reason) })
  })
})
```

## 5. 请求层日志（改造 `useApi.ts`）

把现有"裸 fetch + throw"升级为"带日志的请求"，统一记录请求开始/耗时/失败：

```typescript
// composables/useApi.ts 中统一封装
const log = useLogger('api')

async function request<T>(url: string, init?: RequestInit): Promise<T> {
  const start = performance.now()
  log.info('请求开始', { url, method: init?.method ?? 'GET' })
  try {
    const resp = await fetch(url, init)
    const ms = Math.round(performance.now() - start)
    if (!resp.ok) {
      const err = await resp.json().catch(() => ({ message: `HTTP ${resp.status}` }))
      log.error('请求失败', { url, status: resp.status, ms, message: err.message })
      throw new Error(err.message || `HTTP ${resp.status}`)
    }
    log.info('请求成功', { url, status: resp.status, ms })
    return resp.json()
  } catch (e) {
    log.error('请求异常', { url, error: String(e) })
    throw e
  }
}
```

> 这样所有 `apiFetchData / apiTriggerSync ...` 都自动带日志，无需逐个改。

## 6. 服务端日志（Nitro / `server/`）

服务端用同一套级别约定，输出结构化 JSON 便于检索：

```typescript
// server/utils/logger.ts
export function serverLog(level: string, module: string, message: string, ctx?: object) {
  console.log(JSON.stringify({
    time: new Date().toISOString(), level, module, message, ...ctx,
  }))
}
```

> 进阶可接入 `pino`（高性能结构化日志），但起步阶段标准库足够。

## 7. "一键复制诊断"（与调试能力联动）

错误页和调试面板都提供"复制诊断信息"按钮，内容 = 当前错误 + `getLogBuffer()` 最近日志 + 环境信息（路由、UA、时间）。非程序人员**直接粘贴给 AI** 即可定位问题。详见 [`04-调试能力设计.md`](./04-调试能力设计.md)。

## 8. 给 AI 的日志规范（写进 AGENTS.md）

```markdown
## 日志规范
- 每个 composable/组件顶部：const log = useLogger('模块名')
- 关键流程用 log.info，异常分支用 log.warn，失败用 log.error
- error 必须带 { error: String(e) } 上下文
- 禁止 console.log / console.error，一律用 useLogger
- 不要在日志里打印密码/token/手机号等敏感信息
```
