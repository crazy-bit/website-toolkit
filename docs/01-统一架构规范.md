# 01 · 统一架构规范

> 目标：让 AI 每次生成的代码**结构一致、风格统一、可预测**，从根本上避免"页面一多就乱"。

---

## 1. 技术栈定版（强约束，二选一处必须唯一）

| 类别 | 选型 | 说明 |
|------|------|------|
| 框架 | **Nuxt 3** | 约定式路由 + 自动导入，对非程序人员最友好 |
| UI 组件库 | **Nuxt UI 3**（默认）| 与 Nuxt 深度集成、自带主题、上手最快 |
| UI 备选 | Shadcn Vue | 仅当需要高度自定义视觉时整体切换，**不与 Nuxt UI 混用** |
| 样式 | **Tailwind CSS**（随 UI 库） | 禁止手写大量裸 CSS，优先工具类 |
| 动画 | **GSAP** | 统一通过 `useGsap()` 调用，禁止散落实现 |
| 平滑滚动 | **Lenis** | 统一通过 `useLenis()` 接入 |
| 图标 | **Iconify**（Lucide + MDI） | 统一 `<UIcon name="i-lucide-xxx" />`，禁止引入零散 svg/图片图标 |
| 语言 | **TypeScript** | 全量 TS，禁止新增 `.js` 业务文件 |

> 决策原则：**主流 > 小众**。技术栈越主流，AI 语料越多，生成越稳。

---

## 2. 目录约定（AI 必须严格遵守）

```
project/
├── app.vue                # 根组件（含全局错误捕获、Lenis 初始化）
├── error.vue              # 友好错误页（见 04 调试文档）
├── nuxt.config.ts         # 唯一配置入口
├── app.config.ts          # 主题/品牌色配置
├── AGENTS.md              # ⭐ AI 协作规范（见第 5 节）
├── assets/
│   └── css/main.css       # 仅放全局样式与变量
├── pages/                 # ⭐ 一个文件 = 一个路由，文件名即 URL
│   ├── index.vue
│   └── about.vue          # → /about
├── components/            # 可复用组件，自动导入，无需 import
│   ├── App*.vue           # 布局类（AppSidebar/AppTopBar...）
│   └── 业务组件.vue
├── composables/           # 逻辑与状态，自动导入
│   ├── useLogger.ts       # 日志（见 02）
│   ├── useApi.ts          # 所有后端请求集中此处
│   ├── useGsap.ts         # 动画
│   └── useLenis.ts        # 滚动
├── layouts/
│   └── default.vue        # 默认布局
├── types/
│   └── index.ts           # 集中类型定义
├── server/                # 需要后端时放这里（Nitro）
│   └── api/               # 服务端接口
└── tests/                 # 测试（见 03）
    ├── unit/
    └── e2e/
```

### 铁律（写进 AGENTS.md）

1. **新增页面只能放 `pages/`**，文件名即路由，不手写路由表。
2. **可复用 UI 必须拆成 `components/`**，单文件不超过 ~300 行，超了就拆。
3. **所有逻辑/状态写进 `composables/`**，组件里不堆业务逻辑。
4. **所有网络请求集中在 `composables/useApi.ts`**，组件不直接 `fetch`。
5. **图标只用 Iconify**，动画只用 `useGsap`，滚动只用 `useLenis`。
6. **全 TypeScript**，类型集中在 `types/`。

## 3. 命名规范

| 对象 | 规范 | 示例 |
|------|------|------|
| 页面文件 | 小写中划线 | `pages/user-profile.vue` |
| 组件文件 | 大驼峰 | `components/UserCard.vue` |
| 布局类组件 | `App` 前缀 | `AppSidebar.vue` |
| 视图区块 | `View` 前缀 | `ViewOverview.vue` |
| composable | `use` 前缀小驼峰 | `useDashboard.ts` |
| 类型 | 大驼峰 | `interface UserProfile {}` |
| 常量 | 全大写下划线 | `const API_BASE = ''` |

## 4. 组件分层模型

```
页面 (pages/*.vue)         ← 只负责"组装"：拉数据、放布局、塞组件
   │ 使用
   ▼
区块/组件 (components/*)    ← 负责"展示"：接收 props，发出事件
   │ 调用
   ▼
逻辑层 (composables/*)      ← 负责"思考"：状态、计算、请求
   │ 调用
   ▼
请求层 (useApi.ts)          ← 负责"通信"：唯一的后端出入口
```

> 这套分层让 AI 改动有"边界感"：改样式动组件、改逻辑动 composable、改接口动 useApi，互不串味。

## 5. ⭐ AI 协作规范文件（AGENTS.md / CLAUDE.md / .cursorrules）

这是统一架构能否真正生效的**开关**。放项目根目录，每次 AI 对话自动遵守。建议内容（命令式、带正反例、尽量短）：

```markdown
# 本项目 AI 协作规范

## 技术栈（不要替换）
Nuxt 3 + Nuxt UI 3 + Tailwind + GSAP + Lenis + Iconify + TypeScript

## 必须遵守
- 新页面放 pages/，文件名即路由
- 可复用 UI 拆到 components/，单文件 < 300 行
- 逻辑/状态写 composables/，组件不堆业务逻辑
- 所有请求走 composables/useApi.ts
- 日志用 useLogger()，禁止 console.log
- 图标用 <UIcon name="i-lucide-xxx" />
- 动画用 useGsap()，滚动用 useLenis()
- 全 TypeScript，类型放 types/

## 禁止
- ❌ 引入新的 UI 库 / 动画库 / 图标方案
- ❌ 在组件里直接 fetch
- ❌ 裸用 console.log
- ❌ 新建 .js 业务文件
- ❌ 手写路由表

## 每次改完代码后
1. 运行 `npm run check`
2. 若红灯，读日志定位并修复，不要绕过测试
```

| 级别 | 包含 | 适用 |
