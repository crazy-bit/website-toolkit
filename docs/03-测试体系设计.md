# 03 · 测试体系设计（单测 + Playwright）

> 核心理念转变：**测试不是给非程序人员写的，是给 AI 写、给 AI 跑的"自动验收员"。**
>
> 工作流：你描述功能 → AI 写实现 + 写测试 → 一键运行 → 红灯则 AI 读日志自我修复 → 绿灯才算完成。

---

## 1. 为什么对非程序人员尤其重要

| 痛点 | 测试如何解决 |
|------|--------------|
| 改 A 坏 B、反复出错 | 回归测试：改完一键跑，坏了立刻红灯，不会带病上线 |
| AI 说"改好了"但其实没好 | 测试是客观验收，绿灯才算数，杜绝"幻觉式完成" |
| 不敢迭代 | 有安全网后，大胆改 |

---

## 2. 测试分层与配比

针对**建站场景**，端到端价值远高于函数单测。建议：

```
        E2E（Playwright）  约 70%   ← "页面能打开、能点、流程能走通"
       ┌──────────────┐
      单元测试（Vitest）约 30%      ← 纯逻辑函数 / composable 计算
   ┌────────────────────┐
```

| 类型 | 工具 | 测什么 |
|------|------|--------|
| 单元测试 | **Vitest** | composables 里的纯逻辑、数据转换、工具函数 |
| 端到端 | **Playwright** | 真实浏览器中页面渲染、交互、跳转、无报错 |
| 路由冒烟 | Playwright（模板化） | 自动遍历所有页面，断言能打开、无 JS 错误 |

## 3. 单元测试（Vitest）

### 安装与配置

```bash
npm i -D vitest @nuxt/test-utils @vue/test-utils happy-dom
```

```typescript
// vitest.config.ts
import { defineVitestConfig } from '@nuxt/test-utils/config'

export default defineVitestConfig({
  test: {
    environment: 'nuxt',
    include: ['tests/unit/**/*.spec.ts'],
  },
})
```

### 示例：测一个 composable 的纯逻辑

```typescript
// tests/unit/useHealth.spec.ts
import { describe, it, expect } from 'vitest'
import { calcHealthLevel } from '~/composables/useHealth'

describe('calcHealthLevel', () => {
  it('完成率 >= 0.8 为健康', () => {
    expect(calcHealthLevel(0.9)).toBe('healthy')
  })
  it('完成率 < 0.4 为风险', () => {
    expect(calcHealthLevel(0.2)).toBe('risk')
  })
})
```

> 约定：**纯计算逻辑要从组件里抽到 composable**，才好被单测覆盖（这也呼应架构分层）。

## 4. 端到端测试（Playwright）

### 安装

```bash
npm i -D @playwright/test
npx playwright install --with-deps
```

```typescript
// playwright.config.ts
import { defineConfig } from '@playwright/test'

export default defineConfig({
  testDir: './tests/e2e',
  use: { baseURL: 'http://localhost:3000', trace: 'on-first-retry' },
  webServer: {
    command: 'npm run dev',
    url: 'http://localhost:3000',
    reuseExistingServer: !process.env.CI,
    timeout: 120_000,
  },
})
```

### 示例：核心流程

```typescript
// tests/e2e/home.spec.ts
import { test, expect } from '@playwright/test'

test('首页能正常打开且无控制台错误', async ({ page }) => {
  const errors: string[] = []
  page.on('console', (msg) => { if (msg.type() === 'error') errors.push(msg.text()) })
  page.on('pageerror', (e) => errors.push(String(e)))

  await page.goto('/')
  await expect(page.locator('body')).toBeVisible()
  await expect(page).toHaveTitle(/.+/)            // 标题非空
  expect(errors, `控制台错误：\n${errors.join('\n')}`).toHaveLength(0)
})
```

## 5. ⭐ 路由冒烟测试模板（本工具的杀手锏）

**自动遍历所有 `pages/` 路由**，每个页面都跑一遍"能打开 + 无 JS 报错 + 有内容"。新增页面**零成本**获得基础保护——这是对非程序人员最实用的安全网。

```typescript
// tests/e2e/smoke.routes.spec.ts
import { test, expect } from '@playwright/test'
import fg from 'fast-glob'
import { resolve } from 'node:path'

// 从 pages/ 目录推导所有路由
function discoverRoutes(): string[] {
  const files = fg.sync('pages/**/*.vue', { cwd: resolve(__dirname, '../..') })
  return files.map(f =>
    '/' + f
      .replace(/^pages\//, '')
      .replace(/\.vue$/, '')
      .replace(/index$/, '')
      .replace(/\[(\w+)\]/g, 'test')   // 动态路由用占位值
      .replace(/\/$/, '') || '/',
  )
}

for (const route of discoverRoutes()) {
  test(`冒烟：${route} 能正常打开`, async ({ page }) => {
    const errors: string[] = []
    page.on('pageerror', e => errors.push(String(e)))
    page.on('console', m => { if (m.type() === 'error') errors.push(m.text()) })

    const resp = await page.goto(route)
    expect(resp?.status(), `${route} HTTP 状态`).toBeLessThan(400)
    await expect(page.locator('body')).toBeVisible()
    // body 文本不为空（非白屏）
    expect((await page.locator('body').innerText()).trim().length).toBeGreaterThan(0)
    expect(errors, `${route} 报错：\n${errors.join('\n')}`).toHaveLength(0)
  })
}
```

> 效果：哪天 AI 改崩了某个页面，`npm run check` 立刻红灯并指出是哪个路由。

## 6. 一键命令（隐藏复杂度）

`package.json`：

```json
{
  "scripts": {
    "test:unit": "vitest run",
    "test:e2e": "playwright test",
    "test:smoke": "playwright test tests/e2e/smoke.routes.spec.ts",
    "check": "npm run lint && nuxt typecheck && vitest run && playwright test tests/e2e/smoke.routes.spec.ts"
  }
}
```

- 非程序人员只需记一条：**`npm run check`**（lint + 类型 + 单测 + 路由冒烟）。
- 全绿才算"改完了"。

## 6.5 ESLint —— 把规范从"靠自觉"升级为"机器兜底"

测试验证"功能对不对"，ESLint 验证"写法符不符合规范"。`full` 级别内置 `eslint.config.mjs`（基于 `@nuxt/eslint`），用静态规则**自动拦截**反向校验中发现的真实问题：

| 规则 | 拦截的反模式 | 来源 |
|------|--------------|------|
| `no-console` | 裸用 `console.log`（应走 `useLogger`） | 规范第五条 |
| `no-empty`（不允许空 catch） | `catch {}` 吞掉错误 | maoxianjia 真实问题 |
| `no-restricted-syntax`（禁 fetch/$fetch） | 组件/页面内直接发请求 | maoxianjia 真实问题 |

```js
// eslint.config.mjs（要点）
'no-console': 'error',
'no-empty': ['error', { allowEmptyCatch: false }],
'no-restricted-syntax': ['error',
  { selector: "CallExpression[callee.name='fetch']", message: '请走 useApi.ts' },
  { selector: "CallExpression[callee.name='$fetch']", message: '请走 useApi.ts' },
],
// 例外：composables/useApi.ts 与 server/** 允许直接 fetch；scripts/** 允许 console
```

> 已并入 `npm run check`，AI 改完代码若违反规范会直接红灯，无需人工 review。

## 7. AI 自我验证工作流（写进 AGENTS.md）

```markdown
## 测试工作流（每次改代码必须执行）
1. 实现功能后，为新增/改动的逻辑补单测，为新增页面无需手写——冒烟模板自动覆盖
2. 运行 `npm run check`
3. 若红灯：
   - 打开失败用例信息 + 应用日志（useLogger 缓冲）定位
   - 修复后重跑，直到全绿
4. 禁止用 .skip / 注释掉测试 来"绕过"红灯
```

```yaml
name: check
on: [push, pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20 }
      - run: npm ci
      - run: npx playwright install --with-deps
      - run: npm run check
```
